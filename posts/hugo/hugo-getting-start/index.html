從 hexo 到 hugo

沒想過會再次開始寫 blog, 多事的 2020 太多意外, 覺得需要即時一一進行想做的事.
於是重新開始了 blog, 把想做的事完成, 紀錄, 然後放水流, 繼續…

雖一直使用 hexo, 但脫節的這些年 hexo 也改了不少. 繼續沿用 hexo 還是換個平台重新來過, 評估了許久.
用 從 hexo 到 hugo 餵狗, 棄坑 hexo 原因大多在效能落差大, 及轉移無痛.

個人心路歷程和 从 Hexo 到 Hugo - ouuan的博客 較為接近,
之前使用個人修改過後的 hexi, 和 hexo 已越來越不相容, 偶爾升版都需解決衝突.

自己後端開發工作漸漸從 node.js 轉移到 go lang, 部落格平台轉移似乎也天注定.

草稿

hexo 把放在 source/_draft 的文章視為草稿, 發佈時不會產生該資料夾下的文章.
hugo 則是以文章 Front Matter 中的 draft 屬性判斷該篇文章是否為草稿, 預設為非草稿.

title: "使用 Hugo 建構個人網站與部落格"
date: 2020-10-06T15:08:05+08:00
draft: true
...
---

兩者設計各有優缺, 個人喜歡 hugo 的方式, 文章編輯完直接改設定再發佈即可, 不需要搬移檔案, 操作比較直觀.

Front Matter

hexo 對於 Front Matter 寬容度較高, 並不一定要包在兩個 --- 之間. 而 hugo 的 Front Matter 則必須包含在 --- 之間, 否則會出錯.

發佈到 github.io

hugo 沒有像 hexo 內建 deploy 的指令, 需自行把檔案上傳網站空間. 我使用 GitHub Pages 來放置個人網站.

以下說明假設對 git 已有一定程度熟悉, 關於 git 如何安裝, 相關指令細節不介紹.

發布 script

hugo 產生的靜態網站文件都放在 public 資料夾下面, 基本上直接把所有檔案加入 git, 並 push 上去即可. 於是我寫了簡單的 script 來完成發佈動作.

hugo
cd public
git add .
git commit -m "Updated: `date +'%Y-%m-%d %H:%M:%S'`"
git push -f
cd ..
